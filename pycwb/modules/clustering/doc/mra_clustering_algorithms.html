MRA Clustering Algorithms

Multi-Resolution Analysis (MRA) clustering backends for gravitational-wave burst trigger formation in the PycWB pipeline.

1. Why MRA Clustering?

The Wilson-Daubechies-Meyer (WDM) transform decomposes strain data into multiple time-frequency resolutions (levels). At each resolution, "hot" pixels — time-frequency cells with excess power — are selected. Coarse resolutions capture broadband, long-duration signals; fine resolutions capture narrowband, short transients.

A real gravitational-wave signal may span several resolutions, producing pixel clusters at each level. Clustering each resolution independently (the legacy approach) can fragment a single astrophysical event into multiple triggers. The MRA backends instead pool pixels from all resolutions before clustering, so a real signal is reconstructed as one coherent cluster.

Res. 0 pixels Res. 1 pixels Res. 2 pixels … Res. M pixels → Pooled MRA pixel table → Clustering → FragmentCluster → Likelihood eval

2. Shared Step — Pixel Pooling

Both MRA backends begin with pool_mra_pixel_candidates(pixel_candidates_by_resolution), which flattens per-resolution pixel candidates into a single MRAPixelPool:

  • PixelArrays — concatenated time, frequency, likelihood, rate, and detector-energy arrays from all resolutions.
  • Sidecar metadata — resolution index, WDM level, raw time & frequency (in bins), physical time bounds (time_start/time_stop), and physical frequency bounds (frequency_low/frequency_high) for each pixel.

Physical coordinates are computed from rate (the sampling rate of each resolution): Δt = 1/rate, Δf = rate/2. These physical coordinates enable cross-resolution comparisons — pixels from different levels can be compared in the same time-frequency space.

mra_hdbscan density-based

Uses HDBSCAN (Hierarchical Density-Based Spatial Clustering of Applications with Noise) from scikit-learn. All pooled pixels are embedded into a feature space, and HDBSCAN identifies clusters as regions of high density separated by regions of low density.

Feature Matrix

Each pooled pixel is mapped to a vector of up to 5 dimensions:

  1. Time center — (tstart + tstop) / (2 · time_scale)
  2. Frequency center — (flow + fhigh) / (2 · freq_scale)
  3. Level (optional) — WDM resolution level, zero-based, to separate pixels by scale.
  4. Log-energy (optional) — log(1 + likelihood), normalized by its median, to separate signal from background.
  5. Detector balance (optional) — fraction of total energy in the first detector, to distinguish coherent signals from glitches.

Cluster Extraction

HDBSCAN's fit_predict assigns each pixel a cluster label or −1 (noise). Noise pixels are optionally promoted to singleton clusters if noise_as_singletons=True. Clusters with fewer than min_pixels are discarded.

Key idea: HDBSCAN automatically discovers the number of clusters and is robust to noise. It does not require a fixed distance threshold — instead it builds a hierarchy of density-based clusters and selects the most persistent ones.

mra_weighted_graph graph-based

Builds a weighted adjacency graph over all pooled pixels, then extracts clusters as connected components using scipy's sparse graph routines.

Edge Weights

An edge between pixels i and j is scored by a product of compatibility factors:

  1. Time–frequency overlap — 0.5·(scoretime + scorefreq). Each axis score decays linearly from 1 to 0 as the gap between intervals grows from 0 to the allowed radius.
  2. Detector similarity — cosine similarity between per-detector energy vectors.
  3. Energy similarity — exp(−β · |log(1+Li)−log(1+Lj)|), penalizing large likelihood gaps.
  4. Resolution penalty — exp(−γ · |leveli − levelj|), discouraging edges between very different resolutions.

Edges with weight below min_edge_weight are dropped. Cross-resolution edges can be disabled via enable_cross_resolution_edges=False, in which case only same-resolution pixels connect.

Cluster Extraction

The upper-triangle edges are symmetrized into a sparse CSR adjacency matrix, and scipy.sparse.csgraph.connected_components labels each pixel by its connected component. Components smaller than min_pixels are discarded.

Key idea: The graph formulation makes physical constraints (time/frequency adjacency, detector coherence, energy consistency) explicit and tunable. It is deterministic — the same input always produces the same clusters.

4. Post-Processing — Likelihood & Output

Both backends wrap their clusters in a single FragmentCluster whose metadata spans the union of all contributing resolutions (earliest start, latest stop, lowest flow, highest fhigh, max rate). This FragmentCluster is then passed through finalize_mra_clusters_for_likelihood, which:

  1. Computes SNR and χ² statistics for the merged cluster.
  2. Optionally runs a final defragmentation pass (final_defrag=True) to re-cluster with tighter constraints.
  3. Writes per-lag trigger output to the Parquet catalog.

5. Choosing Between the Two

Prefer mra_hdbscan when…

  • The number of clusters is unknown a priori.
  • You need automatic noise rejection (unclustered pixels).
  • Signals have complex, non-convex shapes in the time-frequency plane (HDBSCAN handles arbitrary cluster geometries).
  • You want a single, well-studied hyperparameter (min_cluster_size) to control granularity.

Prefer mra_weighted_graph when…

  • Reproducibility is paramount — the graph method is deterministic.
  • You want explicit, physically-motivated constraints (time/frequency adjacency, detector coherence, energy consistency).
  • The signal morphology is well-understood and can be encoded in edge thresholds.
  • You need to disable cross-resolution edges to study resolution-by-resolution behaviour.
Computational cost: The graph backend builds O(N²) edges and is dominated by nested loops in Python — acceptable for typical MRA pixel counts (N ≲ few×10³ per lag). HDBSCAN uses scikit-learn's optimized C implementation and scales better to larger pixel sets.

6. YAML Configuration Example

clustering:
  backend: mra_hdbscan          # or mra_weighted_graph

  mra_hdbscan:
    time_scale_seconds: 0.01
    freq_scale_hz: 64.0
    level_weight: 0.5
    log_energy_weight: 0.25
    detector_balance_weight: 0.5
    min_cluster_size: 3
    min_samples: null
    cluster_selection_epsilon: 0.0
    cluster_selection_method: eom
    noise_as_singletons: true
    min_pixels: 2
    final_defrag: false

  mra_weighted_graph:
    time_radius_bins: 2.0
    freq_radius_bins: 3.0
    min_edge_weight: 0.1
    detector_similarity_weight: 0.5
    energy_similarity_weight: 0.25
    energy_balance_wt: 0.5
    resolution_penalty: 0.1
    max_level_gap: 99
    enable_cross_resolution_edges: true
    min_pixels: 2
    final_defrag: false
